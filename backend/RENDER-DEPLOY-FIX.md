# If Render deploy fails (tripoli-explorer-api)

## 1. Check the logs

In **Render Dashboard** → your service **tripoli-explorer-api** → **Logs**.  
The last lines usually show the real error (e.g. missing env var, DB connection, or build failure).

## 2. Set these in Dashboard → Environment

| Variable       | Required | Example / note |
|----------------|----------|----------------|
| **DATABASE_URL** | Yes     | Your Supabase **Session** URI (port 5432). From Supabase → Project Settings → Database → Connection string → URI → Session. |
| **JWT_SECRET**   | Yes     | Auto-generated by Render. If you override, use at least 32 characters (e.g. `openssl rand -base64 32`). |
| **CORS_ORIGIN**  | No      | Defaults to `*`. For production you can set your app URL (e.g. `https://yourapp.com`). |
| **DB_ACCEPT_SELF_SIGNED** | Yes for Supabase | Set to `1` so the DB connection accepts Supabase’s certificate (fixes “self-signed certificate in certificate chain” in logs). |

After changing env vars, use **Manual Deploy** → **Deploy latest commit**.

## 3. Build vs start failure

- **Build failed:** Check the **Build** log (e.g. `npm install` error). Ensure **Root Directory** is `backend`.
- **Start / crash:** Check the **Runtime** log. Often it’s missing **DATABASE_URL** or **JWT_SECRET**, or the app exited (e.g. DB connection failed). Add **DATABASE_URL** and redeploy.

## 4. Code changes that help

The repo was updated so that:

- **CORS_ORIGIN** is no longer required to be set; it defaults to `*` if missing, so the service can start.
- **render.yaml** sets **CORS_ORIGIN** to `*` by default.

Commit and push these changes, then trigger a new deploy on Render.
